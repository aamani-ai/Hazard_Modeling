# Workflows

How we approach work of meaningful scale in this repo — the repeatable *process*, not the specifics of any
one task. The goal: anyone (human or agent) picking up work follows the same decision-making process that
produces good outcomes, instead of re-inventing it each time.

Adapted from `renewablesinfo_org/docs/workflows` and `model-gpr/docs/plans` — reasoned to fit a
**modeling / notebook** project rather than a web platform (see [`feature_workflow.md`](feature_workflow.md)
for what changed and why).

## Index

| Workflow | When to use |
|----------|-------------|
| [`feature_workflow.md`](feature_workflow.md) | Building a new capability from a blank page — e.g. a peril pipeline (hail → solar). Idea → references → architecture → **phased, notebook-first execution**. |

## The one rule that governs everything

A **reference is a raw input, not a blueprint.** A reference says *"this is possible"*; our
`architecture.md` says *"this is what we build, adapted to us."* We never copy-paste a reference — we
extract what's good, name what we'd do differently, and translate it through reasoning. (This is also why
the old `hazard_analysis` repo is treated as *both* a source and an anti-pattern.)
